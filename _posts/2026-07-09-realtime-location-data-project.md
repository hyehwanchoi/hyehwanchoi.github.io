최근 제조 현장의 실시간 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

공개 글이기 때문에 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 구체적인 장비 구성은 적지 않으려고 한다. 대신 내가 실제로 마주했던 문제와 그 문제를 확인하며 배운 점을 중심으로 정리해보려고 한다.

이 글은 멋있게 포장한 프로젝트 소개라기보다는, 실제로 운영 시스템을 보면서 하나씩 확인했던 기록에 가깝다.

---

## 프로젝트를 간단히 설명하면

이 시스템은 장비에 태그를 붙이고, 여러 수집 지점에서 들어오는 신호를 기반으로 장비의 위치를 계산해 사용자에게 제공하는 시스템이다.

흐름을 단순화하면 다음과 같다.

```text
태그가 부착된 장비
  -> 수집 장비
  -> Kafka
  -> 위치 계산/상태 처리
  -> 후속 시스템 전달
  -> 사용자에게 위치 정보 제공
```

내가 들어왔을 당시에는 데이터를 완전히 실시간으로 처리하기보다는, 일정 주기마다 짧은 시간 동안 메시지를 모아서 보내는 방식에 가까웠다.

하지만 위치 데이터는 현재성이 중요하다. 몇 분 전 위치보다 지금 어디에 가까운지가 더 중요하기 때문에, 이 흐름을 실시간 처리에 더 가깝게 바꾸는 것이 주요한 과제였다.

처리해야 하는 데이터 양도 작지 않았다. 많을 때는 1분에 약 140만 건 수준의 데이터가 들어오는 상황을 고려해야 했다. 이 정도 규모에서는 Kafka partition, producer timeout, 로그 한 줄, DB commit 방식까지 모두 문제가 될 수 있었다.

---

## Kafka partition이 1개였던 문제

처음 확인했던 것 중 하나는 Kafka topic과 partition 구성이었다.

topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition이 1개인 상태였다. topic이 여러 개라는 것만으로는 충분하지 않다. 하나의 topic 안에서 병렬 처리를 하려면 partition이 필요하다.

partition이 1개면 해당 topic은 결국 한 줄로 처리된다. consumer를 늘리더라도 하나의 partition은 동시에 여러 consumer가 나누어 처리할 수 없다.

그래서 topic별 partition 수를 늘리는 작업을 검토했고, 실제 운영에서는 여러 topic을 한 번에 확인하고, 없는 topic은 생성하고, 있는 topic은 partition을 늘리는 스크립트도 준비했다.

이 과정에서 다시 확인한 것은 Kafka에서 성능을 볼 때 topic 개수보다 partition 설계가 더 중요하다는 점이었다.

## Producer timeout

운영 중 다음과 같은 형태의 Kafka producer timeout 로그를 봤다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

처음 보면 producer 설정 문제처럼 보일 수 있다. 하지만 이 에러는 메시지가 producer 내부에서 오래 대기하다가 broker로 전송되지 못하고 만료됐다는 뜻이다.

원인은 하나로 단정할 수 없었다.

- partition이 부족해서 특정 partition에 몰릴 수 있다.
- broker 응답이 늦을 수 있다.
- producer 전송량이 broker 처리량보다 많을 수 있다.
- consumer가 뒤에서 밀리고 있을 수 있다.
- batch, linger, buffer 설정이 현재 트래픽과 맞지 않을 수 있다.

그래서 producer timeout을 볼 때 producer 코드만 보지 않고, topic partition, broker 연결, consumer lag, publish 실패 로그를 함께 봐야 한다고 느꼈다.

Kafka 문제는 한쪽에서 발생한 것처럼 보여도 실제 원인은 파이프라인 다른 쪽에 있을 수 있다.

## 수집 시간 파싱 오류

운영 중 수집 시간 값을 파싱하지 못하는 경고 로그도 확인했다.

문제가 된 값은 이런 형태였다.

```text
20260708084711.03
```

기존 로직은 정해진 시간 포맷만 기대하고 있었는데, 실제 데이터에서는 소수점 이하 자릿수가 달라질 수 있었다.

처음에는 특정 formatter를 하나 더 추가하면 될 것처럼 보였다. 하지만 확인해보니 비슷한 파싱 로직이 여러 경로에 흩어져 있었다. 한 곳만 고치면 다른 경로에서 같은 문제가 다시 발생할 수 있었다.

그래서 시간 파싱을 공통 메서드로 정리하고, 여러 입력 포맷을 같은 기준으로 처리하도록 변경했다.

이 문제를 겪으면서 운영 데이터는 문서에 적힌 포맷대로만 들어오지 않는다는 것을 다시 느꼈다. 그리고 같은 의미의 데이터는 여러 곳에서 각자 파싱하지 말고, 가능한 한 공통화해야 한다.

## AOA/BLE 처리 흐름 정리

프로젝트를 보다 보니 비슷한 역할을 하는 코드인데 메서드명이나 처리 흐름이 조금씩 다른 부분이 있었다.

예를 들어 한쪽은 collect message를 만드는 메서드명이 다르고, 다른 쪽은 payload 파싱 이후 null 처리와 검증 위치가 달랐다.

처음에는 동작만 같으면 괜찮다고 생각할 수 있다. 하지만 운영 시스템에서는 이런 작은 차이가 나중에 장애를 분석할 때 혼란을 만든다.

그래서 AOA와 BLE의 collect message 파싱 흐름을 비슷한 구조로 맞추고, 필수값 누락, 알 수 없는 장비, 알 수 없는 수집 지점 같은 케이스를 파싱 단계에서 정리하도록 개선했다.

이 작업은 기능을 새로 만든 것은 아니지만, 코드의 책임을 더 명확히 하는 작업이었다.

## 로그가 안 찍히는 문제

운영 중 특정 로그 파일에 로그가 쌓이지 않는 문제도 확인했다.

처음에는 코드에서 로그를 찍지 않는 문제인지 봤다. 하지만 실제로는 log4j2 설정에서 marker filter와 appender 라우팅이 원하는 방식과 다르게 동작하는 부분이 있었다.

원하는 방향은 명확했다.

- 일반 로그는 프로세스 로그에 남긴다.
- 추적이 필요한 로그는 별도 로그에도 남긴다.
- marker가 있는 로그는 양쪽에 남길 수 있어야 한다.

이 문제를 보면서 로그는 단순히 `logger.info()`를 추가한다고 끝나는 것이 아니라는 걸 느꼈다. 어떤 logger가 어떤 appender에 연결되어 있는지, marker filter가 accept인지 deny인지, additivity가 어떻게 되어 있는지까지 봐야 했다.

특히 대량 데이터 처리 시스템에서는 로그를 많이 찍는 것도 문제지만, 필요한 로그가 엉뚱한 파일에만 찍히는 것도 큰 문제다.

## DB commit 방식

위치 계산 중 floor를 찾지 못하는 경우 tracker 관리 정보를 업데이트하는 로직도 확인했다.

처음에는 일정 건수마다 commit하는 로직이 있었지만, 잔여 건수가 있을 때 commit이 되는지 확인이 필요했다.

예를 들어 200건마다 commit하는 구조라면, 199건만 처리되고 루프가 끝났을 때 이 데이터가 commit되지 않을 수 있다. 기존 시스템에서는 일정 건수 단위 commit뿐 아니라 마지막 잔여 데이터도 commit하는 흐름이 있었다.

그래서 신규 구조에서도 같은 정책이 유지되도록 정리했다.

이 문제는 기능적으로 눈에 잘 보이지 않는다. 하지만 commit 타이밍이 달라지면 운영 데이터가 다르게 남을 수 있다. 기존 시스템을 새 구조로 바꿀 때는 이런 작은 차이를 놓치면 안 된다.

## 기존 시스템과 신규 시스템 비교

이번 프로젝트에서 중요하게 봤던 것 중 하나는 기존 시스템과 신규 시스템의 비즈니스 로직 차이였다.

단순히 코드를 예쁘게 바꾸는 것이 목표가 아니었다. 실시간 처리 구조로 바꾸더라도 기존 위치 계산, 상태 변경, DB update, 메시지 payload의 의미가 달라지면 안 됐다.

그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음과 같은 것들을 확인했다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존에 있던 예외 처리나 fallback이 빠지지 않았는지
- batch commit 이후 잔여 commit 처리가 유지되는지
- 시작 시 offset 처리 방식이 기존과 달라지지 않았는지

이 과정에서 "리팩터링"과 "비즈니스 로직 변경"은 반드시 분리해서 봐야 한다는 것을 다시 느꼈다.

## seekToEnd 처리

Kafka consumer 시작 시점에 과거 메시지를 읽을지, 최신 메시지부터 읽을지도 확인했다.

기존 구조에는 시작 시점에 최신 offset으로 이동하는 처리가 있었다. 신규 구조는 Vert.x Kafka consumer를 사용하면서 이 부분이 명시적으로 보이지 않았다.

`auto.offset.reset=latest`만으로 충분하다고 생각할 수 있지만, 이미 consumer group offset이 저장되어 있으면 그 offset부터 읽는다.

위치 데이터에서는 오래된 메시지를 뒤늦게 처리하는 것이 오히려 현재 상태를 흐릴 수 있다. 그래서 어떤 consumer에 `seekToEnd`가 필요한지 확인하고, partition 할당 이후 최신 offset으로 이동하는 방식을 검토했다.

이 문제를 통해 offset은 단순한 Kafka 설정이 아니라, 데이터의 의미와 직접 연결된다는 것을 배웠다.

## 서버 디스크와 임시 파일 문제

운영 중 서버 디스크 사용률이 높아진 상황도 확인했다.

처음에는 `/tmp` 아래에 Vert.x cache 디렉토리가 많이 보여서 이것이 원인인지 확인했다. 하지만 실제로 용량을 확인해보니 `/tmp`는 크지 않았고, 다른 디렉토리들이 더 큰 비중을 차지하고 있었다.

또 MySQL binary log가 쌓여 있는 것도 확인했다. 이 파일들은 직접 `rm`으로 지우면 안 되고, MySQL 명령으로 purge해야 한다는 점도 다시 확인했다.

이 과정에서 배운 것은 운영 이슈를 감으로 판단하면 안 된다는 점이다.

디스크가 찼다고 해서 바로 눈에 보이는 디렉토리를 지우면 위험하다. `df`, `du`, 큰 파일 검색, 프로세스 사용 여부 확인을 통해 실제 원인을 좁혀야 한다.

## DB 연결 실패

서비스 기동 중 DB connection pool 초기화 실패도 확인했다.

처음에는 애플리케이션 코드 문제처럼 보였지만, 실제로는 DB 접속 자체가 되지 않는 상황이었다. 로컬 socket으로 MySQL에 붙으려다 실패하는 메시지도 확인했다.

이런 문제는 코드만 봐서는 해결되지 않는다.

- 애플리케이션 설정의 DB host가 맞는지
- MySQL이 실제로 떠 있는지
- local socket 접속인지 TCP 접속인지
- 방화벽이나 네트워크 문제가 있는지
- MyBatis/JDBC 초기화가 필요한 순서대로 되었는지

이런 것들을 함께 봐야 했다.

백엔드 개발자는 애플리케이션 내부 코드뿐 아니라, 애플리케이션이 의존하는 DB와 네트워크 상태도 같이 볼 수 있어야 한다는 생각이 들었다.

## CPU가 튀는 Java 프로세스

운영 중 Java 프로세스 하나가 CPU를 많이 쓰는 상황도 확인했다.

처음에는 비즈니스 로직이 무한 루프를 도는 것인지 의심할 수 있었다. 하지만 thread 이름을 확인해보니 Kafka data plane 쪽 thread가 CPU를 많이 쓰고 있었다.

그래서 프로세스 전체만 보는 것이 아니라 thread 단위로 봐야 했다.

- 어떤 Java 프로세스인지 확인한다.
- `top -H`로 CPU를 많이 쓰는 thread를 본다.
- thread id를 16진수로 바꾼다.
- `jstack`에서 해당 thread stack을 확인한다.

이런 방식으로 봐야 실제로 CPU를 쓰는 지점이 Kafka network thread인지, GC인지, 비즈니스 로직인지 구분할 수 있다.

운영 장애를 볼 때는 "Java가 CPU를 많이 쓴다"에서 멈추면 안 되고, 어느 thread가 어떤 일을 하고 있는지까지 내려가야 한다.

## 내가 만든 성과

이번 프로젝트에서 내가 실제로 했거나 정리한 것들을 적어보면 다음과 같다.

- 주기적으로 모아서 처리하던 흐름을 실시간 처리에 더 가깝게 바꾸는 작업을 진행했다.
- Kafka topic/partition 구성을 확인하고, partition 확장과 topic 정리 스크립트를 준비했다.
- producer timeout의 원인을 producer 설정 하나로 단정하지 않고, partition, broker, consumer lag 관점까지 함께 보려고 했다.
- 수집 시간 파싱 오류를 공통 파서로 정리했다.
- AOA/BLE의 메시지 파싱 흐름과 로그 위치를 더 일관되게 맞췄다.
- 기존 시스템과 신규 시스템의 비즈니스 로직 차이를 비교했다.
- floor를 찾지 못하는 경우의 DB commit 정책을 기존 동작과 맞추도록 정리했다.
- 로그 라우팅과 marker 기반 로그 정책을 검토했다.
- 디스크, MySQL binlog, Vert.x cache, DB 연결 실패 같은 운영 이슈도 함께 확인했다.

화려한 기능을 만든 것보다, 운영 시스템에서 실제로 문제가 되는 부분들을 하나씩 확인하고 정리한 경험이 더 컸다.

## 앞으로 해야 할 것들

앞으로는 더 수치 기반으로 확인하고 싶다.

현재는 로그와 운영 명령어를 통해 문제를 좁혀가고 있지만, 장기적으로는 다음 지표를 더 체계적으로 봐야 한다.

- Kafka consumer lag
- producer timeout 건수
- topic별 메시지 유입량
- partition별 메시지 분산 정도
- Java process CPU/GC
- DB update/commit 시간
- 로그 파일 증가량

또한 장애 대응 절차도 문서화해야 한다.

예를 들어 Kafka timeout이 발생했을 때 어떤 로그를 보고, 어떤 명령어로 topic/partition을 확인하고, consumer lag를 어떻게 볼지 정리되어 있으면 대응 시간이 줄어든다.

기존 시스템과 신규 시스템의 동작 차이를 검증하는 테스트도 더 보강하고 싶다. 구조를 바꾸는 작업일수록 결과가 같다는 것을 증명할 수 있어야 한다.

---

## 마무리

이번 프로젝트를 하면서 백엔드 개발의 범위가 생각보다 넓다는 것을 다시 느꼈다.

코드만 고치는 것이 아니라 Kafka, DB, 로그, 서버 디스크, 프로세스 상태까지 함께 봐야 했다. 특히 대용량 실시간 데이터 처리 시스템에서는 작은 설정 하나가 지연이나 장애로 이어질 수 있다.

내가 이번 프로젝트에서 얻은 가장 큰 경험은 문제를 단정하지 않고 끝까지 확인하는 태도였다.

Kafka timeout이 나면 producer만 보는 것이 아니라 partition과 consumer까지 보고, 로그가 안 찍히면 코드만 보는 것이 아니라 log4j2 설정까지 보고, 디스크가 차면 눈에 보이는 파일을 지우기 전에 실제 용량을 확인해야 한다.

앞으로는 이런 경험을 바탕으로, 단순히 기능을 구현하는 개발자보다 운영 가능한 시스템을 만들고 개선하는 백엔드 개발자로 더 성장하고 싶다.
