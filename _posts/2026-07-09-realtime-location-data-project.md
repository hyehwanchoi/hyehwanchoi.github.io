최근 제조 현장의 실시간 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

구체적인 회사명, 내부 시스템명, 서버 구성, topic 이름, 네트워크 정보는 공개 글에 적합하지 않기 때문에 이 글에서는 모두 일반화해서 정리하려고 한다. 대신 어떤 문제를 마주했고, 어떤 관점으로 개선했으며, 그 과정에서 무엇을 배웠는지를 중심으로 남겨보려고 한다.

---

## 프로젝트를 한 문장으로 설명하면

이 프로젝트는 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 장비의 위치를 계산하고, 그 결과를 사용자 또는 후속 시스템에 제공하는 실시간 위치 데이터 처리 시스템이다.

쉽게 말하면 다음과 같은 흐름이다.

```text
태그가 부착된 장비
  -> 수집 장비
  -> 메시지 브로커
  -> 위치 계산/상태 처리
  -> 후속 시스템 전달
  -> 사용자에게 위치 정보 제공
```

현장에서는 장비의 위치 정보가 늦게 반영되면 데이터의 가치가 떨어진다. 몇 분 전 위치보다 지금 어디에 가까운지가 중요하기 때문이다.

그래서 이 시스템에서 중요한 것은 단순히 데이터를 많이 처리하는 것이 아니라, 많은 데이터를 가능한 한 지연 없이 안정적으로 처리하는 것이다.

---

## 처음 마주한 구조

처음 프로젝트에 들어왔을 때는 완전한 실시간 처리라기보다는 일정 시간 단위로 데이터를 모아 처리하는 방식에 가까웠다.

이 방식은 구현이 단순하고 한 번에 묶어서 처리하기 쉽다는 장점이 있다. 하지만 위치 데이터처럼 현재성이 중요한 데이터에는 한계가 있었다.

또한 메시지 topic은 여러 개로 나뉘어 있었지만, partition 구성이 충분하지 않아 특정 구간에서 병렬 처리의 이점을 제대로 활용하기 어려운 상태였다.

데이터 유입량도 작지 않았다. 순간적으로 많은 데이터가 몰릴 수 있고, 전체적으로는 분당 백만 건 이상의 데이터를 고려해야 하는 상황이었다. 이 정도 규모에서는 코드 구조뿐 아니라 Kafka partition, producer 설정, 로그 정책, DB commit 방식까지 모두 성능과 안정성에 영향을 준다.

---

## 개선 방향

내가 잡은 큰 방향은 세 가지였다.

첫째, 데이터를 더 실시간에 가깝게 처리하도록 흐름을 바꾸는 것.

둘째, 대량 데이터 처리에서 병목이 되는 지점을 줄이는 것.

셋째, 장애가 발생했을 때 원인을 추적할 수 있도록 운영성을 높이는 것.

기능을 많이 추가하는 것보다 기존 시스템의 의도를 해치지 않으면서 처리 구조를 안정화하는 것이 더 중요했다. 특히 기존 시스템과 신규 구조의 비즈니스 로직이 달라지면 안 되는 상황이었기 때문에, 변경 전후의 동작을 계속 비교하면서 작업했다.

---

## Kafka에서 겪은 문제

가장 많이 들여다본 부분은 Kafka였다.

대량 메시지를 처리하다 보니 producer timeout이 발생하는 상황이 있었다. 메시지가 broker로 제때 전송되지 못하고 producer 내부에서 오래 대기하다가 만료되는 형태였다.

이런 문제는 producer 설정 하나만 바꾼다고 해결되지 않는다.

- topic partition 수가 충분한지
- 메시지 key가 특정 partition에 몰리지 않는지
- consumer가 유입량을 따라가고 있는지
- broker 응답이 지연되고 있지는 않은지
- producer batch와 buffer 설정이 현재 트래픽에 맞는지

이런 것들을 함께 봐야 했다.

이 과정을 거치면서 Kafka는 단순한 메시지 전달 도구가 아니라, 시스템의 처리량과 지연 시간을 결정하는 중요한 설계 요소라는 것을 다시 느꼈다.

---

## 실시간 시스템에서 offset의 의미

또 하나 중요했던 것은 consumer offset이었다.

Kafka의 `auto.offset.reset=latest` 설정은 이름만 보면 항상 최신 메시지부터 읽을 것 같지만, 실제로는 consumer group에 저장된 offset이 없을 때만 적용된다.

이미 offset이 저장되어 있다면 서비스가 재기동되어도 저장된 offset부터 다시 읽는다.

위치 데이터 시스템에서는 이 동작이 항상 원하는 결과는 아닐 수 있다. 오래된 메시지를 뒤늦게 처리하면 현재 위치를 오염시킬 수 있기 때문이다.

그래서 어떤 consumer는 backlog를 처리해야 하고, 어떤 consumer는 시작 시점의 최신 메시지부터 처리해야 하는지를 구분해서 봐야 했다. 실시간 시스템에서는 "모든 데이터를 빠짐없이 처리하는 것"과 "현재 상태를 정확하게 유지하는 것" 사이에서 선택이 필요한 순간이 있다.

---

## 데이터 포맷 문제

운영 중에는 시간 포맷 문제도 있었다.

수집 시간 값이 항상 같은 자릿수로 들어오지 않았고, 소수점 이하 자릿수가 달라지는 경우가 있었다. 기존 파싱 로직은 정해진 포맷만 기대하고 있어서 일부 데이터에서 경고 로그가 발생했다.

이 문제는 단순히 예외 하나를 처리하는 문제가 아니었다.

같은 시간 값이 여러 경로에서 파싱되고 있었기 때문에, 각 클래스에 흩어진 파싱 로직을 유지하면 앞으로도 같은 문제가 반복될 가능성이 있었다. 그래서 공통 파서로 정리하고, 여러 입력 포맷을 일관되게 처리하도록 변경했다.

작은 수정처럼 보이지만 운영 시스템에서는 이런 종류의 정리가 중요하다. 같은 의미의 값을 여러 방식으로 해석하면 장애가 나도 원인을 찾기 어려워진다.

---

## 로그와 추적성

폐쇄망 환경에서는 외부 모니터링 도구를 자유롭게 붙이기 어렵다.

그래서 로그가 매우 중요하다. 하지만 대량 데이터 처리 경로에 로그를 많이 넣으면 로그 자체가 병목이 된다.

이번 프로젝트에서는 일반 처리 로그와 추적용 로그를 구분하려고 했다.

- 평소에는 필요한 수준의 처리 로그만 남긴다.
- 특정 흐름을 추적해야 할 때는 marker 기반 로그를 남긴다.
- publish 실패, 파싱 실패, 알 수 없는 장비나 수집 지점은 원인을 알 수 있게 기록한다.
- 대량 루프 안에서는 불필요한 문자열 생성과 info 로그를 피한다.

로그는 많을수록 좋은 것이 아니라, 장애가 났을 때 원인을 좁힐 수 있어야 한다.

---

## 운영 중 겪은 문제들

코드 외적인 운영 문제도 있었다.

디스크 사용량이 높아져 서버 접근이나 서비스 동작에 영향을 줄 수 있는 상황을 확인했고, 어떤 디렉토리가 용량을 차지하는지 추적했다. Kafka와 Vert.x가 사용하는 임시 디렉토리, MySQL binary log, 애플리케이션 로그처럼 운영 중 자연스럽게 쌓이는 파일들을 구분해서 봐야 했다.

또 DB 연결 실패나 Kafka topic/partition 설정 문제처럼 애플리케이션 코드만 봐서는 해결되지 않는 이슈도 있었다.

이런 경험을 하면서 백엔드 개발자는 코드만 보는 것이 아니라, 애플리케이션이 실제로 실행되는 환경까지 함께 이해해야 한다는 것을 다시 느꼈다.

---

## 내가 만든 성과

이번 프로젝트에서 내가 만든 성과를 정리하면 다음과 같다.

첫째, 배치성 처리에 가까웠던 흐름을 실시간 처리에 더 적합한 구조로 개선했다.

둘째, Kafka topic/partition 구조와 producer timeout 문제를 분석하면서 대량 메시지 처리 병목을 줄일 수 있는 방향을 잡았다.

셋째, 시간 값 파싱, 메시지 파싱 흐름, 공통 builder, commit 처리 등 중복되거나 흩어진 로직을 정리했다.

넷째, 기존 시스템과 신규 구조의 비즈니스 로직 차이를 비교하면서, 구조를 바꾸더라도 기존 동작이 달라지지 않도록 검토했다.

다섯째, 장애 추적을 위한 로그 기준을 정리하고, 운영 중 어떤 정보를 남겨야 문제를 좁힐 수 있는지 고민했다.

이 과정에서 단순히 기능을 구현하는 것보다, 시스템이 안정적으로 오래 동작하게 만드는 일이 훨씬 어렵고 중요하다는 것을 많이 느꼈다.

---

## 앞으로 해야 할 것들

아직 해야 할 일도 많다.

가장 먼저 필요한 것은 수치 기반의 관찰이다.

현재 처리량, consumer lag, producer timeout, CPU 사용률, GC, DB commit 시간 같은 지표를 더 명확히 봐야 한다. 감으로 "느리다"거나 "괜찮다"고 판단하는 것이 아니라, 수치로 판단할 수 있어야 한다.

또한 장애 상황에서 어떤 순서로 확인해야 하는지 운영 가이드를 정리할 필요가 있다.

예를 들어 Kafka timeout이 발생했을 때는 broker 연결, topic partition, consumer lag, producer 설정, 로그 증가량을 어떤 순서로 볼지 정리해두면 대응 시간이 줄어든다.

테스트도 더 보강해야 한다.

특히 기존 시스템과 신규 구조의 비즈니스 로직이 동일하게 동작하는지 검증하는 테스트가 중요하다. 구조 개선은 내부 구현이 바뀌는 일이기 때문에, 결과가 같다는 것을 확인할 수 있어야 한다.

마지막으로 이 경험을 문서화해야 한다.

시스템 구조, 데이터 흐름, 장애 대응 방법, Kafka 운영 명령어, 로그 확인 방법을 정리해두면 나뿐만 아니라 다음에 이 시스템을 보는 사람도 훨씬 빠르게 이해할 수 있다.

---

## 마무리

이번 프로젝트는 나에게 단순한 기능 개발보다 더 큰 경험이었다.

대량의 실시간 데이터를 처리하는 시스템에서는 코드, Kafka, DB, 로그, 서버 운영이 따로 떨어져 있지 않다. 하나의 문제가 다른 영역의 증상으로 나타나고, 작은 설정 하나가 전체 처리량에 영향을 준다.

10년차 개발자로서 앞으로 더 가져가야 할 방향도 조금 더 선명해졌다.

나는 단순히 API를 만드는 개발자보다, 대량 데이터를 안정적으로 처리하고 운영 가능한 구조로 개선하는 백엔드 개발자로 성장하고 싶다.

이번 프로젝트는 그 방향을 확인하게 해준 좋은 경험이었다.
