최근 제조 현장의 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있습니다.

이 시스템은 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 위치를 계산하고, 그 결과를 후속 시스템과 사용자에게 제공하는 역할을 합니다. 많을 때는 1분에 약 140만 건 수준의 데이터를 고려해야 했고, 외부 서비스를 자유롭게 붙이기 어려운 폐쇄망 환경에서 운영된다는 제약도 있었습니다.

구체적인 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 장비 구성은 공개 글에 적합하지 않기 때문에 모두 일반화해서 작성합니다. 대신 이번 글에서는 다음 내용을 중심으로 정리해보려고 합니다.

- 왜 기존 처리 구조를 바꿔야 했는지
- 실시간 처리로 바꾸며 어떤 원칙을 세웠는지
- Kafka topic/partition과 timeout을 어떻게 바라봤는지
- JMeter로 무엇을 검증하려 했는지
- 코드와 로그는 어떤 기준으로 정리했는지

한 편에 모든 기술적 디테일을 담기보다는, 먼저 전체 흐름을 따라가 보려고 합니다. 이 글은 "이렇게 하면 됩니다"라는 정답지라기보다, 운영 중인 시스템을 개선하며 어떤 문제를 마주했고 어떤 선택을 했는지에 대한 기록에 가깝습니다.

## 1부. 왜 바꿔야 했나

### 기존 처리 방식

처음 시스템을 봤을 때 가장 먼저 확인한 것은 데이터 처리 흐름이었습니다.

기존 구조는 완전한 실시간 처리라기보다, 일정 주기 안에서 일부 데이터를 모아 메시지를 전송하고 처리하는 방식에 가까웠습니다. 예를 들면 10분이라는 주기 안에서 1분간 데이터를 모아 처리하는 식이었습니다.

이 방식은 구현이 단순하고, 일정 단위로 데이터를 묶어 처리하기 쉽다는 장점이 있습니다. 데이터의 현재성이 크게 중요하지 않은 경우라면 충분히 합리적인 선택일 수도 있습니다.

하지만 위치 데이터에서는 이야기가 조금 달랐습니다.

사용자가 보고 싶은 것은 "조금 전 어느 위치였는가"가 아니라 "지금 어디에 가까운가"입니다. 처리 지연이 길어질수록 위치 데이터의 가치는 떨어집니다. 특히 현장 장비의 위치를 판단하는 시스템이라면, 데이터가 늦게 반영되는 순간 사용자에게는 이미 과거 정보가 됩니다.

그래서 이번 개선의 첫 번째 목표는 명확했습니다.

> 일정 시간 데이터를 모아 처리하던 구조를, 메시지를 받는 즉시 처리하는 실시간 흐름으로 바꾸는 것.

### 실시간 전환이 단순하지 않았던 이유

처음에는 consumer가 메시지를 더 자주 읽고 바로 처리하면 되는 것처럼 보일 수 있습니다.

하지만 실제로는 그렇지 않았습니다.

메시지를 실시간으로 받기 시작하면, 그 뒤의 모든 구간도 그 속도를 따라갈 수 있어야 합니다. 위치 계산이 느리면 consumer lag가 쌓이고, 후속 publish가 밀리면 producer timeout이 발생하고, 로그를 너무 많이 남기면 로그 자체가 병목이 될 수 있습니다.

즉, 실시간 처리로 바꾼다는 것은 단순히 "읽는 시점"을 바꾸는 일이 아니었습니다.

- Kafka가 유입량을 받아낼 수 있는지
- consumer가 처리 속도를 따라갈 수 있는지
- 후속 시스템으로 보내는 publish가 밀리지 않는지
- 로그와 DB update가 병목이 되지 않는지
- 기존 비즈니스 로직의 의미가 유지되는지

이 모든 흐름을 함께 봐야 했습니다.

## 2부. 먼저 세운 원칙

구조를 바꾸기 전에 몇 가지 원칙을 먼저 세웠습니다.

### 1. 비즈니스 로직의 의미를 바꾸지 않는다

이번 작업의 목표는 "다르게 동작하는 새 시스템"을 만드는 것이 아니었습니다. 처리 타이밍과 구조는 바뀌더라도 위치 계산 결과, 상태값, 후속 시스템으로 전달되는 payload의 의미는 유지되어야 했습니다.

그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음을 확인했습니다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존 예외 처리나 fallback이 빠지지 않았는지
- commit 타이밍이 운영 데이터에 다른 결과를 만들지 않는지
- 시작 시 offset 처리 방식이 기존 의도와 맞는지

운영 중인 시스템을 개선할 때 가장 어려운 부분은 여기라고 생각합니다. 코드 구조를 바꾸는 것보다, 기존 시스템이 암묵적으로 지키고 있던 동작을 이해하고 보존하는 일이 더 어렵습니다.

### 2. 병목을 추측하지 않는다

Kafka producer timeout이 발생한다고 해서 producer 설정만 바로 바꾸지는 않았습니다.

producer timeout은 producer 자체의 문제일 수도 있지만, partition 부족, key 쏠림, broker 처리량, consumer 지연 같은 문제가 producer 쪽 증상으로 드러난 것일 수도 있습니다.

그래서 이번 작업에서는 Kafka UI, 로그, 성능 테스트를 함께 보면서 병목을 좁혀가려고 했습니다.

### 3. 운영 중 추적 가능한 구조로 만든다

폐쇄망 환경에서는 외부 모니터링 도구를 자유롭게 붙이기 어렵습니다.

그렇기 때문에 코드와 로그가 더 중요했습니다. 장애가 발생했을 때 어느 단계에서 실패했는지, 어떤 메시지 흐름에서 문제가 생겼는지, 어떤 파일에 어떤 로그가 남는지 추적할 수 있어야 했습니다.

## 3부. Kafka 구조부터 보기

실시간 처리에서 가장 먼저 본 것은 Kafka topic과 partition 구성이었습니다.

topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition 수가 충분하지 않은 상태였습니다. topic을 나누는 것만으로는 병렬 처리 한계가 해결되지 않습니다. Kafka에서 실제 병렬 처리의 기본 단위는 partition입니다.

partition이 부족하면 consumer를 늘려도 기대한 만큼 처리량이 늘지 않습니다. 특정 topic이나 partition에 메시지가 몰리고, 그 결과 producer timeout이나 consumer lag로 증상이 나타날 수 있습니다.

그래서 Kafka UI를 통해 다음을 확인했습니다.

- topic 구성
- partition 수
- 메시지 적재 흐름
- consumer group 상태
- 처리 중 밀림이 발생하는 구간

이후 필요한 topic과 partition 설정을 조정했습니다. 이 작업의 목적은 단순히 partition 수를 늘리는 것이 아니라, 실시간 처리 구조에서 병렬 처리 여지를 확보하는 것이었습니다.

### partition을 늘리면 끝일까?

물론 partition을 늘린다고 모든 문제가 해결되지는 않습니다.

partition 수가 늘어도 key가 특정 값에 몰리면 특정 partition만 계속 바빠질 수 있습니다. 또한 partition 수가 늘면 운영해야 할 단위도 늘어납니다. rebalance 비용이나 모니터링 복잡도도 함께 고려해야 합니다.

그래서 partition 조정은 "성능을 위해 많이 늘리자"가 아니라, 현재 데이터 흐름과 병목을 보고 필요한 만큼 조정하는 작업이어야 했습니다.

## 4부. Timeout 로그가 말해준 것

운영 중 다음과 같은 형태의 Kafka producer timeout 로그를 확인했습니다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

이 로그는 producer가 메시지를 broker로 제때 보내지 못하고 내부에서 오래 대기하다가 만료됐다는 의미입니다.

처음 보면 producer 설정을 먼저 의심하게 됩니다. 하지만 실제로 확인해야 할 지점은 더 많았습니다.

- partition이 부족하거나 특정 partition으로 몰리는가
- broker가 현재 유입량을 받아낼 수 있는가
- consumer가 뒤에서 밀리고 있지는 않은가
- producer batch, linger, buffer 설정이 현재 트래픽에 맞는가
- 성능 테스트 중 timeout이 재현되는가

Kafka는 producer, broker, consumer가 연결된 흐름입니다. 한쪽의 지연이 다른 쪽의 timeout으로 나타날 수 있기 때문에 전체 흐름을 함께 봐야 했습니다.

이번 경험을 통해 다시 확인한 것은 단순합니다.

> Kafka 문제는 한 컴포넌트의 문제가 아니라 흐름 전체의 문제로 봐야 한다.

## 5부. JMeter로 검증하기

구조를 바꾼 뒤에는 JMeter를 활용해 성능 테스트를 진행했습니다.

목표는 단순히 요청이 성공하는지 보는 것이 아니었습니다. 실시간 처리 구조가 실제 유입량을 어느 정도까지 감당하는지, Kafka 설정 변경이 효과를 내는지, 병목이 어느 구간에서 발생하는지 확인하는 것이었습니다.

JMeter 테스트에서 중점적으로 본 것은 다음과 같습니다.

- 부하 증가 시 처리 지연이 발생하는가
- Kafka topic 또는 partition에 메시지가 쌓이는가
- producer timeout이 발생하는가
- consumer가 처리 속도를 따라가는가
- 로그 또는 후속 처리 과정이 병목이 되는가

운영 데이터와 완전히 동일한 조건을 만들기는 어렵습니다. 하지만 반복 가능한 부하 테스트 시나리오를 만들면 변경 전후를 비교할 수 있습니다.

실제로 Kafka UI와 로그를 함께 보면서 테스트를 진행하니, 코드만 봤을 때는 알기 어려웠던 지점들이 보였습니다. 어떤 변경은 기대만큼 효과가 있었고, 어떤 부분은 추가 확인이 필요했습니다.

성능 테스트는 마지막에 통과 여부만 확인하는 절차가 아니었습니다. 설계 판단을 보정하는 도구에 가까웠습니다.

## 6부. 코드 구조 정리

실시간 처리 구조로 바꾸면서 코드도 함께 정리했습니다.

기존 코드에는 비슷한 역할을 하는 흐름인데 파일명, 메서드명, 검증 위치, 로그 위치가 조금씩 다른 부분이 있었습니다. 동작만 보면 큰 문제는 아닐 수 있지만, 운영 중 장애를 분석하거나 기능을 수정할 때는 이런 차이가 비용이 됩니다.

그래서 다음 방향으로 정리했습니다.

- 파일명과 메서드명을 더 명확하게 변경
- 메시지 파싱 흐름 통일
- 필수값 검증 위치 정리
- 후속 시스템으로 보낼 payload 생성 흐름 정리
- 로그 위치와 메시지 형식 정리

특히 비슷한 처리 흐름은 다형성을 활용해 공통 구조와 타입별 차이를 분리하려고 했습니다.

공통 흐름은 상위 구조에서 이해할 수 있게 두고, 타입별로 달라지는 부분만 각 구현체에서 처리하도록 만들면 변경 범위가 명확해집니다. 새로운 조건이 추가되더라도 전체 흐름을 다시 읽지 않고 확장 지점만 보면 됩니다.

리팩터링의 목적은 코드를 보기 좋게 만드는 것이 아니었습니다. 실시간 처리 시스템에서 장애를 더 빨리 찾고, 변경을 더 안전하게 하기 위한 정리였습니다.

## 7부. 로그 정책 정리

대량 데이터 처리 시스템에서 로그는 신중하게 다뤄야 합니다.

로그가 너무 많으면 로그 자체가 병목이 됩니다. 반대로 필요한 로그가 없으면 장애를 추적할 수 없습니다.

이번 개선에서는 로그 정책도 함께 정리했습니다.

- 일반 처리 로그와 추적용 로그를 구분
- 특정 흐름을 추적해야 할 때 marker 기반 로그 활용
- 로그가 어느 파일에 남아야 하는지 명확히 정리
- 대량 처리 루프 안의 불필요한 로그 제거
- 실패 로그에는 원인을 좁힐 수 있는 최소한의 식별 정보 포함

폐쇄망 환경에서는 외부 관찰 도구를 자유롭게 붙이기 어렵습니다. 그렇기 때문에 로그는 단순한 디버깅 도구가 아니라 운영 설계의 일부가 됩니다.

## 8부. 앞으로 더 해보고 싶은 것

아직 더 확인하고 싶은 것들이 있습니다.

### Kafka 운영 지표 체계화

Kafka UI로 확인할 수 있는 것도 있지만, 장기적으로는 topic별 메시지 유입량, partition별 offset, consumer group lag, producer 실패 건수를 더 지속적으로 볼 수 있어야 합니다.

### partition key 분산 확인

partition 수를 늘리는 것만으로는 충분하지 않습니다. 메시지 key가 특정 값에 몰리면 결국 특정 partition만 바빠질 수 있습니다. 장비 식별자 기준으로 메시지를 보낼 때 partition별로 데이터가 고르게 분산되는지 확인해야 합니다.

### JMeter 테스트 시나리오 정리

이번에는 주요 흐름을 검증하는 데 집중했다면, 앞으로는 반복 가능한 테스트 시나리오를 더 잘 정리하고 싶습니다. 그래야 Kafka 설정 변경, 코드 리팩터링, 로그 정책 변경이 실제 처리량과 지연 시간에 어떤 영향을 주는지 더 명확히 비교할 수 있습니다.

### 폐쇄망 환경에서의 observability

클라우드 서비스를 쓰지 못하더라도 내부망에서 Prometheus, Grafana, JMX exporter, Kafka exporter 같은 도구를 구성할 수 있는지 검토해볼 수 있습니다. 실제 도입 여부와 별개로 어떤 지표를 봐야 하는지 아는 것은 중요합니다.

## 마무리

이번 프로젝트에서 내가 한 일은 단순히 Kafka consumer를 하나 더 만드는 일이 아니었습니다.

기존의 주기적 수집/처리 구조를 실시간 처리에 더 적합한 흐름으로 바꾸고, Kafka topic과 partition을 조정하고, Kafka UI로 흐름을 모니터링하며, JMeter로 성능 테스트를 진행해 변경 효과를 확인하는 일이었습니다.

동시에 코드 구조를 정리하고, 다형성을 활용해 공통 흐름과 타입별 차이를 분리하고, 파일명과 메서드명을 명확하게 바꾸고, 로그 정책을 운영 관점에서 다시 정리했습니다.

10년차 개발자로서 중요한 것은 단순히 구현을 끝내는 것이 아니라, 시스템이 더 오래 안정적으로 운영될 수 있는 구조를 만드는 것이라고 생각합니다.

이번 글에서는 전체 흐름을 중심으로 정리했습니다. 다음에는 Kafka topic/partition을 조정하면서 어떤 점을 확인했는지, 그리고 JMeter 성능 테스트를 어떻게 구성했는지를 조금 더 자세히 정리해보고 싶습니다.
