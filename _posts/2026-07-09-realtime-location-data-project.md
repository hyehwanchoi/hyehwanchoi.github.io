최근 제조 현장의 실시간 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

공개 글이기 때문에 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 구체적인 장비 구성은 적지 않으려고 한다. 대신 내가 실제로 마주했던 문제와 그 문제를 어떤 관점으로 바라봤는지를 중심으로 정리해보려고 한다.

이 글은 단순한 장애 처리 기록이라기보다, 10년차 백엔드 개발자로서 운영 중인 시스템을 어떻게 개선하고 검증해야 하는지에 대한 기록에 가깝다.

---

## 프로젝트를 간단히 설명하면

이 시스템은 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 위치를 계산하고, 그 결과를 후속 시스템과 사용자에게 제공하는 실시간 위치 데이터 처리 시스템이다.

흐름을 단순화하면 다음과 같다.

```text
태그가 부착된 장비
  -> 수집 장비
  -> Kafka
  -> 위치 계산/상태 처리
  -> 후속 시스템 전달
  -> 사용자에게 위치 정보 제공
```

내가 들어왔을 당시에는 완전한 실시간 처리라기보다, 일정 주기마다 짧은 시간 동안 메시지를 모아서 보내는 방식에 가까웠다. 하지만 위치 데이터는 현재성이 중요하다. 몇 분 전 위치보다 지금 어디에 가까운지가 더 중요하기 때문에, 처리 흐름을 실시간에 더 가깝게 바꾸는 것이 핵심 과제였다.

처리해야 하는 데이터 양도 작지 않았다. 많을 때는 1분에 약 140만 건 수준의 데이터를 고려해야 했다. 이 규모에서는 Kafka partition, producer timeout, 로그 정책, DB commit 방식처럼 평소에는 사소해 보이는 것들도 전체 처리량과 안정성에 영향을 준다.

---

## 실시간 전환에서 먼저 봐야 했던 것

처음 확인한 것은 Kafka topic과 partition 구성이었다.

topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition이 1개인 상태였다. topic이 여러 개라는 것만으로는 병렬 처리 한계가 해결되지 않는다. 하나의 topic 안에서 처리량을 확장하려면 partition 설계가 필요하다.

partition이 1개면 해당 topic은 결국 한 줄로 처리된다. consumer를 늘리더라도 하나의 partition은 동시에 여러 consumer가 나누어 처리할 수 없다.

그래서 topic별 partition 수를 늘리는 작업을 검토했고, 실제 운영에서는 여러 topic을 한 번에 확인하고 정리할 수 있는 스크립트도 준비했다.

이 과정에서 중요했던 것은 "partition을 몇 개로 늘릴 것인가"보다 "현재 병목이 어디에서 생기고 있는가"였다. partition 수, key 분산, consumer 처리량, producer timeout을 함께 봐야 했다.

---

## Producer timeout을 단정하지 않기

운영 중 Kafka producer timeout 로그를 확인했다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

이 로그는 producer가 메시지를 broker로 제때 보내지 못하고 내부에서 오래 대기하다가 만료됐다는 의미다.

처음에는 producer 설정 문제처럼 보일 수 있다. 하지만 실제로는 여러 가능성을 함께 봐야 했다.

- 특정 partition으로 메시지가 몰리는지
- broker 응답이 늦어지는지
- producer 전송량이 broker 처리량을 넘는지
- consumer가 뒤에서 밀리고 있는지
- batch, linger, buffer 설정이 현재 트래픽에 맞는지

Kafka 문제는 한쪽에서 발생한 것처럼 보여도 실제 원인은 파이프라인 다른 쪽에 있을 수 있다. 이 이슈를 보면서 producer timeout을 producer 코드만의 문제로 단정하면 안 된다는 것을 다시 확인했다.

## 데이터 포맷은 생각보다 쉽게 흔들린다

운영 중 수집 시간 값을 파싱하지 못하는 경고 로그도 확인했다.

문제가 된 값은 이런 형태였다.

```text
20260708084711.03
```

기존 로직은 정해진 시간 포맷만 기대하고 있었는데, 실제 데이터에서는 소수점 이하 자릿수가 달라질 수 있었다.

처음에는 formatter를 하나 더 추가하면 될 것처럼 보였다. 하지만 확인해보니 비슷한 시간 파싱 로직이 여러 경로에 흩어져 있었다. 한 곳만 고치면 다른 경로에서 같은 문제가 다시 발생할 수 있었다.

그래서 시간 파싱을 공통 메서드로 정리하고, 여러 입력 포맷을 같은 기준으로 처리하도록 변경했다.

이 작업의 핵심은 단순한 예외 처리 추가가 아니라, 같은 의미의 데이터를 시스템 안에서 일관되게 해석하도록 만든 것이다.

---

## 비슷한 흐름은 비슷하게 보여야 한다

AOA와 BLE 처리 흐름을 보면서 비슷한 역할을 하는 코드인데 메서드명, 검증 위치, 로그 위치가 조금씩 다른 부분을 확인했다.

동작만 같으면 괜찮다고 볼 수도 있다. 하지만 운영 중 장애를 분석할 때 비슷한 흐름이 서로 다른 구조로 되어 있으면 원인을 찾는 시간이 늘어난다.

그래서 collect message 파싱 흐름을 정리하고, 필수값 누락, 알 수 없는 장비, 알 수 없는 수집 지점 같은 케이스를 파싱 단계에서 일관되게 처리하도록 맞췄다.

이 작업은 기능을 새로 만드는 일은 아니었다. 하지만 운영 시스템에서는 이런 정리가 중요하다. 코드의 모양이 비슷해야 장애를 볼 때도 같은 기준으로 판단할 수 있다.

---

## 로그는 기능이 아니라 운영 도구다

운영 중 특정 로그 파일에 로그가 쌓이지 않는 문제도 확인했다.

처음에는 코드에서 로그를 찍지 않는 문제인지 봤지만, 실제로는 log4j2 설정의 marker filter와 appender 라우팅이 원하는 방식과 다르게 동작하는 부분이 있었다.

대량 데이터 처리 시스템에서 로그는 특히 조심해야 한다.

- 너무 많이 찍으면 로그가 병목이 된다.
- 필요한 로그가 없으면 장애를 추적할 수 없다.
- 잘못된 파일에만 찍히면 운영자가 놓칠 수 있다.
- marker와 appender 정책이 명확하지 않으면 로그 의미가 흐려진다.

이번에 로그 라우팅을 보면서 `logger.info()` 한 줄을 추가하는 것보다, 어떤 로그가 어느 파일에 어떤 조건으로 남아야 하는지를 정하는 것이 더 중요하다는 것을 느꼈다.

---

## DB commit 타이밍도 비즈니스 로직이다

위치 계산 중 floor를 찾지 못하는 경우 tracker 관리 정보를 업데이트하는 로직도 확인했다.

기존 시스템에는 일정 건수 단위 commit뿐 아니라 마지막 잔여 데이터도 commit하는 흐름이 있었다. 신규 구조에서 이 부분이 달라지면 기능 화면에서는 바로 드러나지 않더라도 운영 데이터는 다르게 남을 수 있다.

예를 들어 200건마다 commit하는 구조에서 199건만 처리되고 루프가 끝났을 때 잔여 데이터가 commit되지 않으면, 기존 시스템과 신규 시스템의 결과가 달라진다.

그래서 이 부분은 단순한 DB 성능 문제가 아니라 비즈니스 로직 동등성의 문제로 봤다. 구조를 바꾸더라도 commit 정책이 기존 의미를 유지하도록 정리했다.

---

## 기존 시스템과 신규 시스템 비교

이번 프로젝트에서 가장 중요하게 본 것은 기존 시스템과 신규 시스템의 비즈니스 로직 차이였다.

목표는 코드를 예쁘게 바꾸는 것이 아니었다. 실시간 처리 구조로 바꾸더라도 위치 계산, 상태 변경, DB update, 후속 시스템으로 전달되는 payload의 의미가 달라지면 안 됐다.

그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음을 확인했다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존에 있던 예외 처리나 fallback이 빠지지 않았는지
- batch commit 이후 잔여 commit 처리가 유지되는지
- 시작 시 offset 처리 방식이 기존과 달라지지 않았는지

이 과정에서 다시 느낀 것은, 리팩터링과 비즈니스 로직 변경은 반드시 분리해서 봐야 한다는 점이다. 특히 운영 중인 시스템을 실시간 구조로 전환할 때는 "더 좋아 보이는 코드"보다 "같은 결과를 보장하는 구조"가 먼저다.

---

## Offset은 데이터의 의미와 연결된다

Kafka consumer 시작 시점에 과거 메시지를 읽을지, 최신 메시지부터 읽을지도 확인했다.

`auto.offset.reset=latest`만으로 충분하다고 생각할 수 있지만, 이미 consumer group offset이 저장되어 있으면 그 offset부터 읽는다.

위치 데이터에서는 오래된 메시지를 뒤늦게 처리하는 것이 오히려 현재 상태를 흐릴 수 있다. 그래서 어떤 consumer에 `seekToEnd`가 필요한지 확인하고, partition 할당 이후 최신 offset으로 이동하는 방식을 검토했다.

이 문제는 Kafka 설정 문제가 아니라 데이터 의미의 문제였다. 모든 메시지를 빠짐없이 처리하는 것이 중요한 시스템도 있고, 최신 상태를 유지하는 것이 더 중요한 시스템도 있다. 위치 데이터는 그 경계를 명확히 정해야 했다.

---

## 운영 이슈를 보는 방식

프로젝트 중에는 디스크 사용량 증가, MySQL binary log, Vert.x cache, DB connection pool 초기화 실패, Java 프로세스 CPU 상승 같은 운영 이슈도 확인했다.

이 내용들을 블로그에 세부적으로 길게 남기는 것은 크게 중요하지 않다고 생각한다. 10년차 개발자에게 중요한 것은 각각의 명령어를 외우는 것이 아니라, 문제를 단정하지 않고 계층별로 좁혀가는 태도다.

예를 들어 디스크가 찼다고 해서 눈에 보이는 파일부터 지우면 안 된다. DB 연결 실패를 코드 문제로만 보면 안 된다. Java CPU가 높다고 해서 바로 비즈니스 로직 문제라고 단정하면 안 된다.

운영 시스템에서는 증상과 원인이 다른 계층에 있을 수 있다. 그래서 애플리케이션 코드, Kafka, DB, 로그, OS 리소스를 함께 봐야 한다.

## 내가 만든 성과

이번 프로젝트에서 내가 만든 성과를 정리하면 다음과 같다.

- 주기적으로 모아서 처리하던 흐름을 실시간 처리에 더 가깝게 바꾸는 작업을 진행했다.
- Kafka topic/partition 구성을 확인하고, partition 확장과 topic 정리 방안을 준비했다.
- producer timeout을 producer 설정 하나로 단정하지 않고, partition, broker, consumer lag 관점까지 함께 봤다.
- 수집 시간 파싱 오류를 공통 파서로 정리해 여러 경로에서 일관되게 처리하도록 했다.
- AOA/BLE의 메시지 파싱 흐름과 로그 위치를 더 일관되게 맞췄다.
- 기존 시스템과 신규 시스템의 비즈니스 로직 차이를 비교했다.
- floor를 찾지 못하는 경우의 DB commit 정책을 기존 동작과 맞추도록 정리했다.
- 로그 라우팅과 marker 기반 로그 정책을 검토했다.
- 디스크, DB 연결, CPU 상승 같은 운영 이슈를 애플리케이션 관점과 인프라 관점에서 함께 확인했다.

화려한 기능을 새로 만든 것보다, 운영 시스템에서 실제로 문제가 되는 부분들을 하나씩 확인하고 구조적으로 정리한 경험이 더 컸다.

## 앞으로 더 확인하고 싶은 것들

앞으로는 더 수치 기반으로 시스템을 보고 싶다.

현재는 로그와 운영 명령어를 통해 문제를 좁혀가고 있지만, 장기적으로는 다음 지표를 더 체계적으로 봐야 한다.

- Kafka consumer lag
- producer timeout 건수
- topic별 메시지 유입량
- partition별 메시지 분산 정도
- Java process CPU/GC
- DB update/commit 시간
- 로그 파일 증가량

이 시스템은 완전한 폐쇄망 환경에서 운영된다. 그래서 클라우드 기반 모니터링 서비스나 외부 SaaS를 바로 붙이는 방식은 현실적인 선택지가 아니다.

그렇다면 폐쇄망 안에서 가능한 관찰성과 운영 체계를 만들어야 한다.

추가로 검토하고 싶은 것은 다음과 같다.

첫째, Kafka 운영 지표를 더 잘 볼 수 있는 구조를 만드는 것이다.

최소한 topic별 메시지 유입량, partition별 offset, consumer group lag, producer 실패 건수는 빠르게 확인할 수 있어야 한다. 외부 서비스를 쓰지 못하더라도 Kafka CLI, JMX, 내부망에서 운영 가능한 모니터링 도구를 조합할 수 있다.

둘째, partition key 분산을 확인하는 것이다.

partition 수를 늘리는 것만으로는 충분하지 않다. 메시지 key가 특정 값에 몰리면 결국 특정 partition만 바빠질 수 있다. 장비 식별자 기준으로 메시지를 보낼 때 partition별로 데이터가 고르게 분산되는지 확인해야 한다.

셋째, producer와 consumer 처리 시간을 실제 트래픽 기준으로 측정하는 것이다.

`batch.size`, `linger.ms`, `buffer.memory`, `delivery.timeout.ms`, `acks` 같은 producer 설정은 처리량과 안정성에 직접적인 영향을 준다. 동시에 consumer 한 건 처리 시간이 길어지면 lag가 증가한다. JSON 파싱, 위치 계산, DB update, Kafka publish 중 어디에서 시간이 쓰이는지 구간별로 측정해야 한다.

넷째, 장애 대응 runbook을 만드는 것이다.

Kafka timeout, 로그 누락, DB 연결 실패, 디스크 사용량 증가, Java CPU 상승 같은 문제는 다시 발생할 수 있다. 각각 어떤 순서로 확인할지 정리해두면 대응 시간을 줄일 수 있다.

다섯째, 기존 시스템과 신규 시스템의 동작 차이를 검증하는 테스트를 더 늘리는 것이다.

구조 개선을 하더라도 결과가 달라지면 안 되는 부분들이 있다. 상태값, payload 필드, fallback 처리, commit 시점 같은 부분은 테스트로 고정해두는 것이 필요하다.

## 개인적으로 더 공부하고 싶은 것들

이번 프로젝트를 하면서 내가 더 공부해야겠다고 느낀 것들도 있다.

첫째, Kafka를 더 깊게 공부해야 한다.

단순히 producer와 consumer를 구현하는 수준을 넘어, partition assignment, rebalance, offset commit 전략, producer 내부 batch 처리, broker replication, ISR 같은 개념을 더 정확히 이해하고 싶다.

둘째, JVM 운영을 더 공부해야 한다.

CPU가 튀는 Java 프로세스를 볼 때 thread dump, GC log, heap 사용량, JMX 지표를 더 자연스럽게 읽을 수 있어야 한다. 백엔드 개발자에게 JVM은 코드 실행 환경이 아니라 운영 분석 도구이기도 하다.

셋째, 온프레미스 환경에서의 observability를 공부하고 싶다.

클라우드 서비스를 쓰지 못하더라도 Prometheus, Grafana, JMX exporter, Kafka exporter 같은 도구를 폐쇄망 안에서 구성할 수 있는지 검토해볼 수 있다. 실제 도입 여부와 별개로, 어떤 지표를 봐야 하는지 아는 것만으로도 문제를 보는 눈이 달라질 것 같다.

넷째, 성능 테스트 방법을 공부하고 싶다.

운영 데이터와 똑같은 환경을 만들 수는 없더라도, Kafka 메시지 유입량을 재현하고 consumer 처리량을 측정하는 작은 부하 테스트 환경은 필요하다. 그래야 설정 변경 전후의 효과를 숫자로 비교할 수 있다.

다섯째, 실시간 위치 데이터 도메인 자체도 더 이해하고 싶다.

단순히 메시지를 빠르게 처리하는 것과 정확한 위치를 제공하는 것은 다르다. 수집 신호의 품질, 수집 지점 정보, floor 판단, fallback 정책이 최종 위치 결과에 어떤 영향을 주는지 더 이해해야 좋은 개선을 할 수 있을 것 같다.

클라우드로 옮길 수 없는 환경이라고 해서 배울 것이 적은 것은 아니다. 오히려 제한된 환경에서 Kafka, JVM, DB, 로그, 서버 운영을 직접 봐야 하기 때문에 백엔드 개발자로서 배울 것이 많다.

---

## 마무리

이번 프로젝트를 하면서 백엔드 개발의 범위가 생각보다 넓다는 것을 다시 느꼈다.

코드만 고치는 것이 아니라 Kafka, DB, 로그, 서버 리소스, 프로세스 상태까지 함께 봐야 했다. 특히 대용량 실시간 데이터 처리 시스템에서는 작은 설정 하나가 지연이나 장애로 이어질 수 있다.

내가 이번 프로젝트에서 얻은 가장 큰 경험은 문제를 단정하지 않고 끝까지 확인하는 태도였다.

Kafka timeout이 나면 producer만 보는 것이 아니라 partition과 consumer까지 보고, 로그가 안 찍히면 코드만 보는 것이 아니라 log4j2 설정까지 보고, 디스크가 차면 눈에 보이는 파일을 지우기 전에 실제 용량을 확인해야 한다.

앞으로는 이런 경험을 바탕으로, 단순히 기능을 구현하는 개발자보다 운영 가능한 시스템을 만들고 개선하는 백엔드 개발자로 더 성장하고 싶다.
