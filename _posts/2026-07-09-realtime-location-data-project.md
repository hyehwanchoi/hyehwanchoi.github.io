최근 제조 현장의 실시간 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

공개 글이기 때문에 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 구체적인 장비 구성은 적지 않으려고 한다. 대신 내가 어떤 문제를 마주했고, 어떤 기준으로 구조를 개선했는지 중심으로 정리해보려고 한다.

이 글은 단순한 작업 기록이 아니라, 10년차 백엔드 개발자로서 운영 중인 시스템을 어떻게 더 실시간에 가깝고, 추적 가능하고, 유지보수 가능한 구조로 바꿔갔는지에 대한 기록이다.

---

## 프로젝트를 간단히 설명하면

이 시스템은 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 위치를 계산하고, 그 결과를 후속 시스템과 사용자에게 제공하는 실시간 위치 데이터 처리 시스템이다.

흐름을 단순화하면 다음과 같다.

```text
태그가 부착된 장비
  -> 수집 장비
  -> Kafka
  -> 위치 계산/상태 처리
  -> 후속 시스템 전달
  -> 사용자에게 위치 정보 제공
```

처리해야 하는 데이터 양은 작지 않았다. 많을 때는 1분에 약 140만 건 수준의 데이터를 고려해야 했다. 이 정도 규모에서는 코드 구조뿐 아니라 Kafka partition, topic 구성, 로그 정책, consumer 처리 방식까지 모두 성능과 운영 안정성에 영향을 준다.

---

## 기존 구조의 한계

내가 들어왔을 당시에는 데이터를 완전히 실시간으로 처리하는 구조가 아니었다.

기존 방식은 일정 주기 동안 일부 데이터를 모은 뒤 메시지를 전송하고 처리하는 흐름에 가까웠다. 예를 들면 10분이라는 주기 안에서 1분간 데이터를 모아 처리하는 방식이었다.

이 방식은 구현이 단순할 수 있지만, 위치 데이터에는 한계가 있었다. 위치 데이터는 현재성이 중요하다. 사용자가 보고 싶은 것은 몇 분 전 위치가 아니라 지금 장비가 어디에 가까운지다.

그래서 내가 잡은 가장 큰 개선 방향은 명확했다.

> 모아서 처리하는 구조를, 메시지를 받는 즉시 처리하는 실시간 흐름으로 바꾸는 것.

이 전환은 단순히 Kafka consumer 코드를 바꾸는 작업이 아니었다. 기존 비즈니스 로직의 의미를 유지하면서 처리 타이밍과 흐름을 바꾸는 작업이었다.

---

## 실시간 처리로 전환하며 중요하게 본 것

실시간 처리로 바꾼다고 해서 무조건 빨라지는 것은 아니다.

메시지를 즉시 받더라도 consumer가 처리하지 못하면 lag가 쌓이고, producer가 보내지 못하면 timeout이 발생한다. 그래서 Kafka topic과 partition 구성을 함께 봐야 했다.

처음 확인했을 때는 topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition 수가 충분하지 않은 상태였다. topic이 많다는 것만으로 병렬 처리가 보장되지는 않는다. Kafka에서 실제 병렬 처리의 기본 단위는 partition이다.

그래서 Kafka UI를 통해 topic, partition, 메시지 흐름을 확인하고, 필요한 topic과 partition 설정을 조정했다. 단순히 설정값을 바꾼 것이 아니라, 현재 데이터 흐름에서 병목이 어디에 생기는지를 확인하면서 조정하려고 했다.

이 과정에서 성능 테스트도 함께 진행했다. 실시간 처리 구조가 실제 유입량을 감당할 수 있는지, partition 수를 늘렸을 때 처리 흐름이 어떻게 달라지는지, timeout이나 lag가 줄어드는지 확인하는 것이 필요했다.

---

## Kafka 문제를 보는 관점

운영 중 Kafka producer timeout 로그를 확인한 적이 있다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

이 로그는 producer가 메시지를 broker로 제때 보내지 못하고 내부에서 오래 대기하다가 만료됐다는 의미다.

10년차 개발자라면 여기서 producer 설정만 바로 바꾸기보다, 전체 흐름을 봐야 한다고 생각한다.

- partition이 부족해 특정 구간에 병목이 생기는지
- key 분산이 적절한지
- broker가 받아낼 수 있는 처리량인지
- consumer가 뒤에서 밀리고 있지는 않은지
- producer batch와 buffer 설정이 현재 트래픽에 맞는지

Kafka 문제는 producer, broker, consumer 중 한 곳에서만 발생하지 않는다. 한쪽의 지연이 다른 쪽의 timeout으로 보일 수 있다. 그래서 Kafka UI와 운영 로그를 함께 보면서 병목 지점을 좁혀가는 방식이 중요했다.

---

## 코드 리팩터링의 기준

이번 프로젝트에서 리팩터링도 중요한 작업이었다.

단순히 코드를 짧게 만들거나 보기 좋게 바꾸는 것이 목적은 아니었다. 실시간 처리 구조에서는 비슷한 흐름이 여러 곳에 흩어져 있으면 장애 분석과 변경이 어려워진다.

그래서 메시지 파싱, 유효성 검증, 후속 시스템으로 보낼 payload 생성, 로그 위치 같은 부분을 더 일관된 구조로 정리했다.

특히 비슷한 역할을 하는 처리 흐름은 다형성을 활용해 공통 구조를 만들고, 각 타입별로 달라지는 부분만 분리하려고 했다. 이렇게 하면 공통 흐름은 한 곳에서 이해할 수 있고, 타입별 차이는 명확한 확장 지점으로 남길 수 있다.

리팩터링에서 중요하게 본 기준은 세 가지였다.

첫째, 기존 비즈니스 로직의 의미를 바꾸지 않을 것.

둘째, 공통 흐름과 타입별 차이를 분리할 것.

셋째, 장애가 났을 때 어느 단계에서 실패했는지 추적 가능하게 만들 것.

이 기준이 없으면 리팩터링은 쉽게 자기만족이 된다. 운영 중인 시스템에서는 코드가 예뻐지는 것보다, 변경 이후에도 같은 결과를 보장하고 문제를 더 빨리 찾을 수 있는지가 더 중요하다.

---

## 이름을 명확하게 하는 것

파일명과 메서드명을 더 명확하게 정리한 것도 의미 있는 작업이었다.

운영 시스템에서는 이름이 곧 문서 역할을 한다. 어떤 파일이 어떤 책임을 가지는지, 어떤 메서드가 어떤 단계의 처리를 담당하는지 이름만 봐도 어느 정도 알 수 있어야 한다.

특히 AOA와 BLE처럼 비슷하지만 완전히 같지는 않은 흐름에서는 이름이 모호하면 변경 범위가 흐려진다. 비슷한 책임은 비슷한 이름으로 맞추고, 실제로 다른 책임은 다른 이름으로 분리해야 한다.

이런 정리는 작은 작업처럼 보이지만, 프로젝트가 커질수록 효과가 커진다. 나중에 장애가 났을 때 누군가 코드를 처음 보더라도 흐름을 따라갈 수 있어야 하기 때문이다.

---

## 로그 정책 정리

실시간 시스템에서 로그는 매우 조심해야 한다.

데이터가 많을 때는 로그 한 줄도 비용이 된다. 반대로 필요한 로그가 없으면 장애를 추적할 수 없다.

그래서 로그 정책을 다시 봤다.

- 일반 처리 로그와 추적용 로그를 구분한다.
- 특정 흐름을 추적해야 할 때는 marker 기반 로그를 활용한다.
- 로그가 어느 파일에 남아야 하는지 명확히 한다.
- 대량 처리 루프 안에서는 불필요한 로그를 줄인다.
- 실패 로그에는 원인을 좁힐 수 있는 최소한의 식별 정보를 남긴다.

이 작업을 하면서 로그는 기능 개발의 부가물이 아니라 운영 설계의 일부라는 것을 다시 느꼈다.

특히 폐쇄망 환경에서는 외부 모니터링 도구를 자유롭게 붙이기 어렵다. 결국 내부 로그와 제한된 모니터링 도구만으로 문제를 찾아야 한다. 그렇기 때문에 로그 정책은 더 중요하다.

---

## 기존 시스템과 신규 시스템의 동등성

실시간 처리 구조로 전환하면서 가장 조심했던 것은 기존 시스템과 신규 시스템의 비즈니스 로직 차이였다.

처리 방식은 바뀌어도 결과의 의미가 바뀌면 안 된다.

그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음을 확인했다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존에 있던 예외 처리나 fallback이 빠지지 않았는지
- commit 타이밍이 운영 데이터에 다른 결과를 만들지 않는지
- 시작 시 offset 처리 방식이 기존 의도와 맞는지

10년차 개발자에게 중요한 것은 단순히 새로운 구조를 만드는 것이 아니라, 기존 시스템이 가지고 있던 암묵적인 동작까지 이해하고 옮기는 것이라고 생각한다.

---

## 성능 테스트와 모니터링

Kafka UI를 활용해 topic, partition, 메시지 흐름을 확인하고 성능 테스트를 진행했다.

성능 테스트에서 보고 싶었던 것은 단순히 "처리가 된다"가 아니었다.

- 실시간 처리 구조에서 메시지가 밀리지 않는지
- partition 수 변경이 처리량에 어떤 영향을 주는지
- 특정 topic이나 partition에 부하가 몰리지 않는지
- producer timeout이 줄어드는지
- consumer lag가 안정적으로 유지되는지

운영 시스템의 성능 개선은 감으로 판단하면 안 된다. 변경 전후를 비교할 수 있는 지표가 있어야 한다.

아직 더 보강해야 할 부분도 있다. 현재는 Kafka UI와 로그를 중심으로 보고 있지만, 앞으로는 consumer lag, producer 실패 건수, partition별 유입량, JVM CPU/GC 같은 지표를 더 체계적으로 수집하고 싶다.

---

## 앞으로 더 확인하고 싶은 것들

이 시스템은 완전한 폐쇄망 환경에서 운영된다. 그래서 클라우드 기반 모니터링 서비스나 외부 SaaS를 바로 붙이는 방식은 현실적인 선택지가 아니다.

그렇다면 폐쇄망 안에서 가능한 관찰성과 운영 체계를 만들어야 한다.

추가로 검토하고 싶은 것은 다음과 같다.

첫째, Kafka 운영 지표를 더 잘 볼 수 있는 구조를 만드는 것이다.

Kafka UI만으로 확인할 수 있는 것도 있지만, 장기적으로는 topic별 메시지 유입량, partition별 offset, consumer group lag, producer 실패 건수를 더 체계적으로 볼 수 있어야 한다.

둘째, partition key 분산을 확인하는 것이다.

partition 수를 늘리는 것만으로는 충분하지 않다. 메시지 key가 특정 값에 몰리면 결국 특정 partition만 바빠질 수 있다. 장비 식별자 기준으로 메시지를 보낼 때 partition별로 데이터가 고르게 분산되는지 확인해야 한다.

셋째, producer와 consumer 처리 시간을 실제 트래픽 기준으로 측정하는 것이다.

`batch.size`, `linger.ms`, `buffer.memory`, `delivery.timeout.ms`, `acks` 같은 producer 설정은 처리량과 안정성에 직접적인 영향을 준다. 동시에 consumer 한 건 처리 시간이 길어지면 lag가 증가한다. 위치 계산, DB update, Kafka publish 중 어디에서 시간이 쓰이는지 구간별로 측정해야 한다.

넷째, 장애 대응 runbook을 만드는 것이다.

Kafka timeout, 로그 누락, DB 연결 실패, 디스크 사용량 증가, Java CPU 상승 같은 문제는 다시 발생할 수 있다. 각각 어떤 순서로 확인할지 정리해두면 대응 시간을 줄일 수 있다.

다섯째, 기존 시스템과 신규 시스템의 동작 차이를 검증하는 테스트를 더 늘리는 것이다.

구조 개선을 하더라도 결과가 달라지면 안 되는 부분들이 있다. 상태값, payload 필드, fallback 처리, commit 시점 같은 부분은 테스트로 고정해두는 것이 필요하다.

---

## 개인적으로 더 공부하고 싶은 것들

이번 프로젝트를 하면서 내가 더 공부해야겠다고 느낀 것도 있다.

첫째, Kafka를 더 깊게 공부해야 한다.

단순히 producer와 consumer를 구현하는 수준을 넘어, partition assignment, rebalance, offset commit 전략, producer 내부 batch 처리, broker replication, ISR 같은 개념을 더 정확히 이해하고 싶다.

둘째, JVM 운영을 더 공부해야 한다.

CPU가 튀는 Java 프로세스를 볼 때 thread dump, GC log, heap 사용량, JMX 지표를 더 자연스럽게 읽을 수 있어야 한다. 백엔드 개발자에게 JVM은 코드 실행 환경이 아니라 운영 분석 도구이기도 하다.

셋째, 폐쇄망 환경에서의 observability를 공부하고 싶다.

클라우드 서비스를 쓰지 못하더라도 Prometheus, Grafana, JMX exporter, Kafka exporter 같은 도구를 내부망에서 구성할 수 있는지 검토해볼 수 있다. 실제 도입 여부와 별개로, 어떤 지표를 봐야 하는지 아는 것만으로도 문제를 보는 눈이 달라질 것 같다.

넷째, 성능 테스트 방법을 더 체계화하고 싶다.

운영 데이터와 똑같은 환경을 만들 수는 없더라도, Kafka 메시지 유입량을 재현하고 consumer 처리량을 측정하는 작은 부하 테스트 환경은 필요하다. 그래야 설정 변경 전후의 효과를 숫자로 비교할 수 있다.

## 마무리

이번 프로젝트에서 내가 한 일은 단순히 Kafka consumer를 하나 더 만드는 일이 아니었다.

기존의 주기적 수집/처리 구조를 실시간 처리에 더 적합한 흐름으로 바꾸고, Kafka topic과 partition을 조정하고, Kafka UI로 흐름을 모니터링하며, 성능 테스트로 변경 효과를 확인하는 일이었다.

동시에 코드 구조를 정리하고, 다형성을 활용해 공통 흐름과 타입별 차이를 분리하고, 파일명과 메서드명을 명확하게 바꾸고, 로그 정책을 운영 관점에서 다시 정리했다.

10년차 개발자로서 중요한 것은 단순히 구현을 끝내는 것이 아니라, 시스템이 더 오래 안정적으로 운영될 수 있는 구조를 만드는 것이라고 생각한다.

앞으로는 이 경험을 바탕으로 대용량 실시간 데이터를 안정적으로 처리하고, 운영 가능한 시스템으로 개선하는 백엔드 개발자로 더 성장하고 싶다.
