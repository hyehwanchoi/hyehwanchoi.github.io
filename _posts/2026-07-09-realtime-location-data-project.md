최근 제조 현장의 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

이 시스템은 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 위치를 계산하고, 그 결과를 후속 시스템과 사용자에게 제공한다. 많을 때는 1분에 약 140만 건 수준의 데이터를 고려해야 했고, 폐쇄망 환경에서 운영된다는 제약도 있었다.

구체적인 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 장비 구성은 공개 글에 적합하지 않기 때문에 모두 일반화해서 작성한다. 대신 이 글에서는 어떤 문제를 해결하려 했고, 어떤 원칙으로 구조를 바꿨고, 어떻게 검증했는지를 중심으로 정리해보려고 한다.

---

## 1. 왜 바꿔야 했나

기존 구조는 완전한 실시간 처리라기보다, 일정 주기 안에서 일부 데이터를 모아 메시지를 전송하고 처리하는 방식에 가까웠다.

예를 들면 10분이라는 주기 안에서 1분간 데이터를 모아 처리하는 흐름이었다. 이 방식은 구현이 단순하고, 일정 단위로 데이터를 묶어 처리하기 쉽다는 장점이 있다.

하지만 위치 데이터에서는 이 방식이 한계를 가진다.

사용자가 보고 싶은 것은 "조금 전 어느 위치였는가"가 아니라 "지금 어디에 가까운가"다. 데이터가 처리되기까지의 지연이 커질수록 위치 데이터의 가치는 떨어진다.

그래서 이번 개선의 첫 번째 목표는 명확했다.

> 일정 시간 데이터를 모아 처리하던 구조를, 메시지를 받는 즉시 처리하는 실시간 흐름으로 바꾸는 것.

단순히 consumer를 더 자주 돌리는 문제가 아니었다. 실시간으로 받기 시작하면 Kafka, 위치 계산, 후속 publish, 로그, DB update까지 전체 흐름이 그 속도를 따라갈 수 있어야 했다.

---

## 2. 개선 원칙

실시간 구조로 전환하면서 몇 가지 원칙을 먼저 세웠다.

첫째, 기존 비즈니스 로직의 의미를 바꾸지 않는다.

처리 방식은 바뀌어도 위치 계산 결과, 상태값, 후속 시스템으로 전달되는 payload의 의미가 달라지면 안 된다. 구조 개선과 비즈니스 로직 변경은 분리해서 봐야 했다.

둘째, 병목을 추측하지 않고 관찰한다.

Kafka producer timeout이 발생한다고 해서 producer 설정만 문제라고 단정하지 않았다. topic, partition, key 분산, consumer lag, broker 처리량을 함께 봐야 했다.

셋째, 운영 중 추적 가능한 구조로 만든다.

폐쇄망 환경에서는 외부 모니터링 도구를 자유롭게 붙이기 어렵다. 그렇기 때문에 로그 정책, 파일명, 메서드명, 처리 흐름이 운영자가 따라갈 수 있는 형태여야 했다.

넷째, 변경 효과를 성능 테스트로 확인한다.

"실시간으로 바꿨다"는 말만으로는 충분하지 않다. 실제 부하를 주었을 때 메시지가 밀리지 않는지, timeout이 줄어드는지, partition 조정이 효과가 있는지 확인해야 했다.

---

## 3. Kafka 구조부터 점검하기

가장 먼저 Kafka topic과 partition 구성을 확인했다.

topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition 수가 충분하지 않은 상태였다. topic을 나누는 것만으로는 병렬 처리 한계가 해결되지 않는다. Kafka에서 실제 병렬 처리의 기본 단위는 partition이다.

partition이 부족하면 consumer를 늘려도 기대한 만큼 처리량이 늘지 않는다. 특정 topic이나 partition에 메시지가 몰리고, 그 결과 producer timeout이나 consumer lag로 증상이 나타날 수 있다.

그래서 Kafka UI를 통해 다음을 확인했다.

- topic 구성
- partition 수
- 메시지 적재 흐름
- consumer group 상태
- 처리 중 밀림이 발생하는 구간

이후 필요한 topic과 partition 설정을 조정했다. 이 작업의 목적은 단순히 partition 수를 늘리는 것이 아니라, 실시간 처리 구조에서 병렬 처리 여지를 확보하는 것이었다.

---

## 4. Producer timeout을 바라보는 방식

운영 중 다음과 같은 형태의 Kafka producer timeout 로그를 확인했다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

이 로그는 producer가 메시지를 broker로 제때 보내지 못하고 내부에서 오래 대기하다가 만료됐다는 의미다.

이때 producer 설정만 바로 조정하는 것은 위험하다. producer timeout은 producer 자체의 문제일 수도 있지만, 전체 파이프라인의 병목이 producer 쪽 증상으로 드러난 것일 수도 있다.

확인해야 할 지점은 다음과 같았다.

- partition이 부족하거나 특정 partition으로 몰리는가
- broker가 현재 유입량을 받아낼 수 있는가
- consumer가 뒤에서 밀리고 있지는 않은가
- producer batch, linger, buffer 설정이 현재 트래픽에 맞는가
- 성능 테스트 중 timeout이 재현되는가

Kafka는 producer, broker, consumer가 연결된 흐름이다. 한쪽의 지연이 다른 쪽의 timeout으로 나타날 수 있기 때문에 전체 흐름을 함께 봐야 했다.

## 5. JMeter로 성능 테스트하기

구조를 바꾼 뒤에는 JMeter를 활용해 성능 테스트를 진행했다.

목표는 단순히 요청이 성공하는지 보는 것이 아니었다. 실시간 처리 구조가 실제 유입량을 어느 정도까지 감당하는지, Kafka 설정 변경이 효과를 내는지, 병목이 어느 구간에서 발생하는지 확인하는 것이었다.

JMeter 테스트에서 중점적으로 본 것은 다음과 같다.

- 부하 증가 시 처리 지연이 발생하는가
- Kafka topic 또는 partition에 메시지가 쌓이는가
- producer timeout이 발생하는가
- consumer가 처리 속도를 따라가는가
- 로그 또는 후속 처리 과정이 병목이 되는가

운영 데이터와 완전히 동일한 조건을 만들기는 어렵다. 하지만 반복 가능한 부하 테스트 시나리오를 만들면 변경 전후를 비교할 수 있다.

실제로 Kafka UI와 로그를 함께 보면서 테스트를 진행하니, 코드만 봤을 때는 알기 어려웠던 지점들이 보였다. 어떤 변경은 기대만큼 효과가 있었고, 어떤 부분은 추가 확인이 필요했다.

이 과정에서 성능 테스트는 마지막 검증 단계가 아니라, 설계 판단을 보정하는 도구라는 생각이 들었다.

---

## 6. 코드 구조 정리

실시간 처리 구조로 바꾸면서 코드도 함께 정리했다.

기존 코드에는 비슷한 역할을 하는 흐름인데 파일명, 메서드명, 검증 위치, 로그 위치가 조금씩 다른 부분이 있었다. 동작만 보면 큰 문제는 아닐 수 있지만, 운영 중 장애를 분석하거나 기능을 수정할 때는 이런 차이가 비용이 된다.

그래서 다음 방향으로 정리했다.

- 파일명과 메서드명을 더 명확하게 변경
- 메시지 파싱 흐름 통일
- 필수값 검증 위치 정리
- 후속 시스템으로 보낼 payload 생성 흐름 정리
- 로그 위치와 메시지 형식 정리

특히 비슷한 처리 흐름은 다형성을 활용해 공통 구조와 타입별 차이를 분리하려고 했다.

공통 흐름은 상위 구조에서 이해할 수 있게 두고, 타입별로 달라지는 부분만 각 구현체에서 처리하도록 만들면 변경 범위가 명확해진다. 새로운 조건이 추가되더라도 전체 흐름을 다시 읽지 않고 확장 지점만 보면 된다.

리팩터링의 목적은 코드를 보기 좋게 만드는 것이 아니었다. 실시간 처리 시스템에서 장애를 더 빨리 찾고, 변경을 더 안전하게 하기 위한 정리였다.

---

## 7. 로그 정책 정리

대량 데이터 처리 시스템에서 로그는 신중하게 다뤄야 한다.

로그가 너무 많으면 로그 자체가 병목이 된다. 반대로 필요한 로그가 없으면 장애를 추적할 수 없다.

이번 개선에서는 로그 정책도 함께 정리했다.

- 일반 처리 로그와 추적용 로그를 구분
- 특정 흐름을 추적해야 할 때 marker 기반 로그 활용
- 로그가 어느 파일에 남아야 하는지 명확히 정리
- 대량 처리 루프 안의 불필요한 로그 제거
- 실패 로그에는 원인을 좁힐 수 있는 최소한의 식별 정보 포함

폐쇄망 환경에서는 외부 관찰 도구를 자유롭게 붙이기 어렵다. 그렇기 때문에 로그는 단순한 디버깅 도구가 아니라 운영 설계의 일부가 된다.

---

## 8. 기존 동작을 지키기

이번 프로젝트에서 가장 조심했던 부분은 기존 시스템과 신규 시스템의 동작 차이였다.

실시간 처리 구조로 전환하더라도 결과의 의미가 바뀌면 안 된다. 그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음을 확인했다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존 예외 처리나 fallback이 빠지지 않았는지
- commit 타이밍이 운영 데이터에 다른 결과를 만들지 않는지
- 시작 시 offset 처리 방식이 기존 의도와 맞는지

운영 중인 시스템을 개선할 때 어려운 점은 여기에 있다. 겉으로 보이는 구조를 바꾸는 것보다, 기존 시스템이 암묵적으로 지키고 있던 동작을 이해하고 보존하는 일이 더 어렵다.

---

## 9. 앞으로 더 해보고 싶은 것

아직 더 확인하고 싶은 것들이 있다.

첫째, Kafka 운영 지표를 더 체계적으로 보고 싶다.

Kafka UI로 확인할 수 있는 것도 있지만, 장기적으로는 topic별 메시지 유입량, partition별 offset, consumer group lag, producer 실패 건수를 더 지속적으로 볼 수 있어야 한다.

둘째, partition key 분산을 더 자세히 확인하고 싶다.

partition 수를 늘리는 것만으로는 충분하지 않다. 메시지 key가 특정 값에 몰리면 결국 특정 partition만 바빠질 수 있다. 장비 식별자 기준으로 메시지를 보낼 때 partition별로 데이터가 고르게 분산되는지 확인해야 한다.

셋째, JMeter 기반 성능 테스트를 더 체계화하고 싶다.

이번에는 주요 흐름을 검증하는 데 집중했다면, 앞으로는 반복 가능한 테스트 시나리오를 더 잘 정리하고 싶다. 그래야 Kafka 설정 변경, 코드 리팩터링, 로그 정책 변경이 실제 처리량과 지연 시간에 어떤 영향을 주는지 더 명확히 비교할 수 있다.

넷째, 폐쇄망 환경에서 가능한 observability를 고민하고 싶다.

클라우드 서비스를 쓰지 못하더라도 내부망에서 Prometheus, Grafana, JMX exporter, Kafka exporter 같은 도구를 구성할 수 있는지 검토해볼 수 있다. 실제 도입 여부와 별개로 어떤 지표를 봐야 하는지 아는 것은 중요하다.

---

## 마무리

이번 프로젝트에서 내가 한 일은 단순히 Kafka consumer를 하나 더 만드는 일이 아니었다.

기존의 주기적 수집/처리 구조를 실시간 처리에 더 적합한 흐름으로 바꾸고, Kafka topic과 partition을 조정하고, Kafka UI로 흐름을 모니터링하며, JMeter로 성능 테스트를 진행해 변경 효과를 확인하는 일이었다.

동시에 코드 구조를 정리하고, 다형성을 활용해 공통 흐름과 타입별 차이를 분리하고, 파일명과 메서드명을 명확하게 바꾸고, 로그 정책을 운영 관점에서 다시 정리했다.

10년차 개발자로서 중요한 것은 단순히 구현을 끝내는 것이 아니라, 시스템이 더 오래 안정적으로 운영될 수 있는 구조를 만드는 것이라고 생각한다.

앞으로는 이 경험을 바탕으로 대용량 실시간 데이터를 안정적으로 처리하고, 운영 가능한 시스템으로 개선하는 백엔드 개발자로 더 성장하고 싶다.
