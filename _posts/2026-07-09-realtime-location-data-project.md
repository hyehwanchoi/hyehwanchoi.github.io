최근 제조 현장의 실시간 위치 데이터 처리 시스템을 개선하는 프로젝트를 진행하고 있다.

공개 글이기 때문에 회사명, 내부 시스템명, 서버명, IP, 실제 topic 이름, 구체적인 장비 구성은 적지 않으려고 한다. 대신 내가 어떤 문제를 마주했고, 어떤 판단으로 구조를 바꿨고, 어떻게 검증하려고 했는지를 중심으로 정리해보려고 한다.

돌이켜보면 이 프로젝트는 "Kafka를 붙였다"는 이야기가 아니었다. 이미 운영 중인 시스템의 처리 방식을 바꾸고, 그 변화가 실제로 더 나은지 성능 테스트로 확인하고, 다시 운영 가능한 코드와 로그로 다듬어가는 과정이었다.

---

## 처음 본 시스템

이 시스템은 장비에 부착된 태그의 신호를 여러 수집 지점에서 받아 위치를 계산하고, 그 결과를 후속 시스템과 사용자에게 제공한다.

흐름을 단순화하면 다음과 같다.

```text
태그가 부착된 장비
  -> 수집 장비
  -> Kafka
  -> 위치 계산/상태 처리
  -> 후속 시스템 전달
  -> 사용자에게 위치 정보 제공
```

처음 시스템을 봤을 때 가장 먼저 눈에 들어온 것은 처리 방식이었다.

기존 구조는 데이터를 완전히 실시간으로 처리하기보다는, 일정 주기 안에서 일부 데이터를 모아 메시지를 전송하고 처리하는 방식에 가까웠다. 예를 들면 10분이라는 주기 안에서 1분간 데이터를 모아 처리하는 식이었다.

이 방식이 항상 나쁜 것은 아니다. 일정 시간 동안 데이터를 모아 처리하면 구현은 단순해질 수 있다. 하지만 이 시스템은 위치 데이터를 다룬다. 위치 데이터에서 중요한 것은 "언젠가 처리되는 것"이 아니라 "지금의 상태를 얼마나 빠르게 반영하는가"다.

그때부터 내가 봐야 할 방향은 비교적 분명해졌다.

> 모아서 처리하던 흐름을, 메시지를 받는 즉시 처리하는 실시간 흐름으로 바꿔야 한다.

---

## 실시간으로 바꾼다는 것

처음에는 단순히 consumer가 메시지를 더 자주 읽게 만들면 되는 것처럼 보일 수 있다.

하지만 실제로는 그렇게 간단하지 않았다.

메시지를 실시간으로 받기 시작하면, 그 뒤의 모든 구간도 그 속도를 따라갈 수 있어야 한다. 위치 계산이 느리면 밀리고, Kafka publish가 느리면 timeout이 나고, 로그를 너무 많이 찍으면 로그가 병목이 된다.

처리해야 하는 데이터 양도 작지 않았다. 많을 때는 1분에 약 140만 건 수준의 데이터를 고려해야 했다. 이 정도 규모에서는 "일단 동작한다"와 "운영 가능하다" 사이의 차이가 꽤 크다.

그래서 실시간 전환은 코드 변경 하나가 아니라 전체 흐름을 다시 보는 작업이었다.

- Kafka topic과 partition은 충분한가
- consumer는 병렬 처리 이점을 활용하고 있는가
- producer timeout은 어디서 발생하는가
- 로그는 필요한 만큼만 남고 있는가
- 기존 비즈니스 로직의 의미는 유지되는가
- 변경 효과를 숫자로 확인할 수 있는가

이 질문들이 이번 작업의 기준이 되었다.

---

## Kafka를 먼저 봤다

실시간 처리에서 가장 먼저 본 것은 Kafka였다.

topic은 여러 개로 나뉘어 있었지만, 각 topic의 partition 수가 충분하지 않은 상태였다. topic이 많다는 것만으로 병렬 처리가 보장되지는 않는다. Kafka에서 실제 병렬 처리의 기본 단위는 partition이다.

partition이 부족하면 consumer를 늘려도 처리량이 기대만큼 늘지 않는다. 결국 특정 topic이나 partition에 메시지가 몰리고, producer나 consumer 한쪽에서 병목이 드러난다.

그래서 Kafka UI를 통해 topic, partition, 메시지 흐름을 확인했다. 어떤 topic이 존재하는지, partition이 어떻게 잡혀 있는지, 메시지가 어느 정도 쌓이는지 보면서 조정 지점을 찾았다.

이 과정에서 topic과 partition 설정도 정리했다. 필요한 topic은 생성하고, partition 수가 부족한 topic은 늘리고, 변경 후에는 Kafka UI로 다시 확인했다.

Kafka 설정을 바꾸는 일 자체는 어렵지 않다. 중요한 것은 왜 바꾸는지다. 단순히 partition을 늘리는 것이 아니라, 실시간 처리 구조에서 병렬 처리 여지를 확보하고 병목을 줄이기 위한 변경이어야 했다.

---

## Timeout 로그가 알려준 것

운영 중 Kafka producer timeout 로그를 확인했다.

```text
TimeoutException: Expiring records ... ms has passed since batch creation
```

이 로그를 처음 보면 producer 설정을 먼저 의심하게 된다. 하지만 10년차 개발자라면 여기서 한 걸음 더 봐야 한다고 생각했다.

producer가 메시지를 못 보내는 이유는 producer 내부에만 있지 않을 수 있다. 특정 partition에 몰렸을 수도 있고, broker가 받아내지 못했을 수도 있고, consumer가 뒤에서 밀리면서 전체 파이프라인이 불안정해졌을 수도 있다.

그래서 timeout을 볼 때는 producer 설정, partition 수, key 분산, broker 상태, consumer lag를 함께 봐야 했다.

이번 프로젝트에서 Kafka를 보며 가장 크게 느낀 것은 이것이다.

> Kafka 문제는 한 컴포넌트의 문제가 아니라 흐름 전체의 문제로 봐야 한다.

---

## JMeter로 확인하기

구조를 바꿨다면 그 다음은 검증이다.

"실시간으로 바꿨다"는 말만으로는 부족하다. 실제로 메시지가 많이 들어왔을 때 처리할 수 있는지, 병목이 어디서 생기는지, Kafka 설정을 바꾼 것이 효과가 있는지 확인해야 한다.

그래서 JMeter를 활용해 성능 테스트를 진행했다.

JMeter를 통해 실제 운영 데이터와 완전히 동일한 환경을 만들 수는 없지만, 일정 수준의 부하를 재현하고 시스템이 어떻게 반응하는지 확인할 수 있었다.

성능 테스트에서 보고 싶었던 것은 단순히 성공/실패가 아니었다.

- 메시지 유입량이 증가할 때 처리 지연이 생기는가
- Kafka topic이나 partition에 메시지가 쌓이는가
- producer timeout이 발생하는가
- consumer가 처리 속도를 따라가는가
- 로그나 DB update 같은 부가 작업이 병목이 되는가

성능 테스트를 해보면 추측이 줄어든다.

코드만 볼 때는 괜찮아 보이던 부분도 부하를 주면 다르게 보인다. 반대로 막연히 위험하다고 생각했던 부분이 실제로는 병목이 아닐 수도 있다. JMeter로 부하를 만들고 Kafka UI와 로그를 함께 보면서, 변경 전후를 더 구체적으로 비교할 수 있었다.

이 과정은 나에게 꽤 중요했다.

10년차 개발자에게 필요한 것은 "내가 이렇게 바꿨다"에서 끝나는 것이 아니라, "이 변경이 어떤 조건에서 어떤 효과를 내는지 확인했다"까지 가는 것이라고 생각한다.

---

## 코드도 흐름에 맞게 정리했다

실시간 처리 구조로 바꾸면서 코드도 함께 정리했다.

비슷한 역할을 하는 코드가 서로 다른 이름과 흐름을 가지고 있으면 운영 중 장애를 분석하기 어렵다. 특히 AOA와 BLE처럼 비슷하지만 완전히 같지는 않은 흐름에서는 더 그렇다.

그래서 메시지 파싱, 유효성 검증, 후속 시스템으로 보낼 payload 생성, 로그 위치를 더 일관된 구조로 정리했다.

이 과정에서 다형성을 활용해 공통 흐름과 타입별 차이를 분리하려고 했다. 공통 흐름은 상위 구조에서 이해할 수 있게 두고, 각 타입마다 달라지는 부분은 명확한 확장 지점으로 남기는 방식이다.

이렇게 하면 새로운 조건이 추가되거나 특정 타입의 처리를 바꿔야 할 때 변경 범위가 명확해진다.

리팩터링의 기준은 세 가지였다.

첫째, 기존 비즈니스 로직의 의미를 바꾸지 않을 것.

둘째, 공통 흐름과 타입별 차이를 분리할 것.

셋째, 장애가 났을 때 어느 단계에서 실패했는지 추적 가능하게 만들 것.

운영 중인 시스템에서는 코드가 예뻐지는 것보다, 변경 이후에도 같은 결과를 보장하고 문제를 더 빨리 찾을 수 있는지가 더 중요하다.

---

## 이름을 바꾸는 것도 설계다

파일명과 메서드명도 더 명확하게 정리했다.

이름은 작아 보이지만, 운영 시스템에서는 이름이 문서 역할을 한다. 어떤 파일이 어떤 책임을 가지는지, 어떤 메서드가 어떤 단계의 처리를 담당하는지 이름만 봐도 어느 정도 알 수 있어야 한다.

비슷한 책임은 비슷한 이름으로 맞추고, 실제로 다른 책임은 다른 이름으로 분리했다.

이런 정리는 당장 성능을 올리지는 않는다. 하지만 장애가 났을 때 코드를 처음 보는 사람이 흐름을 따라갈 수 있게 해준다. 10년차 개발자가 해야 하는 리팩터링은 이런 부분까지 포함한다고 생각한다.

---

## 로그 정책을 다시 봤다

대량 데이터 처리 시스템에서 로그는 양날의 검이다.

로그가 너무 많으면 로그 자체가 병목이 된다. 반대로 필요한 로그가 없으면 장애를 추적할 수 없다.

그래서 로그 정책을 다시 정리했다.

- 일반 처리 로그와 추적용 로그를 구분한다.
- 특정 흐름을 추적해야 할 때는 marker 기반 로그를 활용한다.
- 로그가 어느 파일에 남아야 하는지 명확히 한다.
- 대량 처리 루프 안에서는 불필요한 로그를 줄인다.
- 실패 로그에는 원인을 좁힐 수 있는 최소한의 식별 정보를 남긴다.

특히 이 시스템은 폐쇄망 환경에서 운영된다. 외부 모니터링 도구를 자유롭게 붙이기 어렵기 때문에 로그의 역할이 더 크다.

그래서 로그는 기능 개발의 부가물이 아니라 운영 설계의 일부로 봐야 했다.

---

## 기존 동작을 지키는 일

이번 프로젝트에서 가장 조심했던 것은 기존 시스템과 신규 시스템의 비즈니스 로직 차이였다.

처리 방식은 바뀌어도 결과의 의미가 바뀌면 안 된다.

그래서 기존 프로젝트와 신규 프로젝트를 비교하면서 다음을 확인했다.

- 같은 상황에서 같은 상태값을 넣는지
- 같은 데이터 필드를 후속 시스템에 전달하는지
- 기존에 있던 예외 처리나 fallback이 빠지지 않았는지
- commit 타이밍이 운영 데이터에 다른 결과를 만들지 않는지
- 시작 시 offset 처리 방식이 기존 의도와 맞는지

실시간 구조로 전환하는 것은 단순히 더 빠른 구조를 만드는 일이 아니다. 기존 시스템이 암묵적으로 지키고 있던 동작을 이해하고, 필요한 것은 유지하고, 바꿔야 하는 것은 명확히 바꾸는 일이다.

---

## 앞으로 더 확인하고 싶은 것들

아직 더 해보고 싶은 것들이 있다.

첫째, Kafka 운영 지표를 더 체계적으로 보고 싶다.

Kafka UI로 확인할 수 있는 것도 있지만, 장기적으로는 topic별 메시지 유입량, partition별 offset, consumer group lag, producer 실패 건수를 더 지속적으로 볼 수 있어야 한다.

둘째, partition key 분산을 더 자세히 확인하고 싶다.

partition 수를 늘리는 것만으로는 충분하지 않다. 메시지 key가 특정 값에 몰리면 결국 특정 partition만 바빠질 수 있다. 장비 식별자 기준으로 메시지를 보낼 때 partition별로 데이터가 고르게 분산되는지 확인해야 한다.

셋째, JMeter 기반 성능 테스트를 더 체계화하고 싶다.

운영 데이터와 완전히 같은 조건을 만들 수는 없더라도, 반복 가능한 부하 테스트 시나리오가 있어야 한다. 그래야 Kafka 설정 변경, 코드 리팩터링, 로그 정책 변경이 실제 처리량과 지연 시간에 어떤 영향을 주는지 비교할 수 있다.

넷째, 폐쇄망 환경에서 가능한 observability를 고민하고 싶다.

클라우드 서비스를 쓰지 못하더라도 내부망에서 Prometheus, Grafana, JMX exporter, Kafka exporter 같은 도구를 구성할 수 있는지 검토해볼 수 있다. 실제 도입 여부와 별개로 어떤 지표를 봐야 하는지 아는 것은 중요하다.

다섯째, 장애 대응 runbook을 만들고 싶다.

Kafka timeout, 로그 누락, DB 연결 실패, 디스크 사용량 증가, Java CPU 상승 같은 문제는 다시 발생할 수 있다. 각각 어떤 순서로 확인할지 정리해두면 대응 시간을 줄일 수 있다.

## 마무리

이번 프로젝트에서 내가 한 일은 단순히 Kafka consumer를 하나 더 만드는 일이 아니었다.

기존의 주기적 수집/처리 구조를 실시간 처리에 더 적합한 흐름으로 바꾸고, Kafka topic과 partition을 조정하고, Kafka UI로 흐름을 모니터링하며, JMeter로 성능 테스트를 진행해 변경 효과를 확인하는 일이었다.

동시에 코드 구조를 정리하고, 다형성을 활용해 공통 흐름과 타입별 차이를 분리하고, 파일명과 메서드명을 명확하게 바꾸고, 로그 정책을 운영 관점에서 다시 정리했다.

10년차 개발자로서 중요한 것은 단순히 구현을 끝내는 것이 아니라, 시스템이 더 오래 안정적으로 운영될 수 있는 구조를 만드는 것이라고 생각한다.

앞으로는 이 경험을 바탕으로 대용량 실시간 데이터를 안정적으로 처리하고, 운영 가능한 시스템으로 개선하는 백엔드 개발자로 더 성장하고 싶다.
