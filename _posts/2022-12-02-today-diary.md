오늘은 회사 업무를 하면서 실수에 대해 고백하고 반성하는 글을 작성해보려고 합니다.  
"다시는 같은 실수를 반복하지 말자" 라는 생각에 이 글을 작성합니다.  
  
제가 했던 실수는 코드 리팩토링을 했는데 적용 전 날 테스트가 되어있지 않았고,  
프로세스 파악이 되어있지 않아 원복을 하게 되었습니다.  
이로 인해 업무를 할당해줬던 파트장님에게 신뢰를 잃었습니다.  
  
같은 실수를 반복하지 않기 위해 항상 다음과 같은 생각을 가지고 업무를 해야겠습니다.  
```
1. 간단한 업무(코드 리팩토링)라도 히스토리와 프로세스를 파악할 것
2. 기능이 잘 동작하는 지 테스트 해 볼 것(API 요청 등...)
3. 항상 의심할 것(잘 될것이라는 믿음 X, 나 자신부터 의심 해 볼 것)
4. 성공 뿐만 아니라 실패 케이스도 테스트 해 볼 것
```

다른 사람은 어떻게 개발 업무를 진행하는 지 궁금해서 검색해보니 이 **[블로그](https://engineer-mole.tistory.com/285)** 에서는 다음과 같이 제시를 하고 있습니다.  

```
1. 요청 사항 파악하기.
2. 업무의 배경이나 유스케이스도 확인하기.
3. 현재 로직은 어떻게 되어있는지 확인하기.
4. 데이터양, 성능 목표도 확인하기.
5. 테스트 코드의 유무도 확인하기.
6. 기능 추가 또는 변경하기 위해 어떻게 해야하는 지 먼저 분석하기.
7. 각 작업의 소요시간 대중해보기.
8. 작업이 끝나면 리뷰 받기.
9. 구현하기.
10. 잘 안된다면 도움 요청하기.
```

제가 빠트렸던 내용들을 채울 수 있어서 이 블로그 글이 참 마음에 드네요.  
오늘을 잊지 않고 다시는 실수하지 않는 개발자가 되어야겠습니다.
